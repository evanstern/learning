Three ways to unstick the GPU

Who gets the 8GB card?

  • Reclaim it
    take the GPU from Emby outright; Emby drops to CPU-only transcoding
  • Restructure the sharing
    move Emby off exclusive PCI passthrough? onto an LXC container?, which can share the GPU concurrently
  • Stay CPU-only for now
    skip the GPU fight entirely, run inference on the Ryzen's 8 cores instead
Home Lab LLM Hosting · the options

The decision

Restructure via LXC sharing

Buys flexibility: Emby can eventually get its own small transcode card, and/or the 4060 Ti gets replaced with something bigger — without undoing today's decision.

Real cost, not a free lunch: a full VM's passthrough is exclusive by design. Sharing the card means Emby itself moves from a VM to an LXC container — not just a config flag.

Which engine

Ollama, or closer to the metal?

  • Ollama — hides the mechanics
    dead simple, sane defaults, great if the goal is "just have a working assistant"
  • llama.cpp / vLLM — closer to the metal

Two different philosophies

llama.cpp vs. vLLM

llama.cpp

  • Flexible GPU/CPU split via n_gpu_layers?
  • Uses GGUF? quantization
  • Built for one user on constrained, shared hardware

vLLM

  • Built around PagedAttention? + continuous batching
  • Wants the model to fit cleanly in VRAM; weak at partial offload
  • Built for many concurrent requests hitting one model

The decision

llama.cpp, for now

Today's constraints — 8GB, shared with Emby, one user — point straight at llama.cpp. The future "AI hub" vision is actually closer to what vLLM is built for.

Deliberately chose to optimize for learning-now over building-for-scale-later. Reversible later: it's a runtime swap, not an architecture commitment.

What actually happens under contention

No graceful fallback — and the fix

Misconception: VRAM doesn't gracefully "spill" into system RAM at runtime. llama.cpp's GPU/CPU split is fixed at model load — running out of VRAM mid-generation (e.g. a long paste colliding with Emby's KV cache?) is a hard CUDA out-of-memory crash, not a slowdown.
Fix: hard-cap the max context length so peak KV cache + model weights stay under ~7GB — enforced inside llama.cpp itself, since consumer GPUs have no OS-level VRAM partitioning to lean on.

What this locks in — and what it doesn't

Foreclosed vs. open

Foreclosed (for now)

  • Top-end reasoning on sprawling, multi-file tasks
  • Emby's own VRAM headroom — now shared, not exclusive

Stays open

  • Swapping models later is just a new GGUF file — no rearchitecting
  • The upgrade path (dedicated Emby card, bigger GPU) is completely untouched

Fits the pattern already in place

One more isolated service, not a special case

  • Every existing service already gets its own VM/LXC
    RAM and CPU capped per-service, specifically so one can't swamp another (downloads vs. Emby)
  • The new LLM LXC follows the exact same rule
    isolation isn't a new idea being introduced — it's the house style
  • RAM is already tight — 160GB configured against 124GB physical
    being actively addressed via a broader VM cleanup, not ignored
  • More non-GPU services on deck
    Vaultwarden, Linkwarden, downloads, the *arr stack — none of them compete for the 8GB card

Key takeaways

What to walk away with

  • The real constraint was never "does the hardware exist" — it was who gets the GPU.
  • Every design choice here traded peak performance for headroom and reversibility — a deliberate pattern, not indecision.
  • VRAM beats RAM on speed because of bandwidth, not because RAM is broken — and that same fact explains why running out of it fails hard, not gracefully.
  • Deferring the hardware upgrade didn't close it off — the mechanical room to grow was checked and confirmed before parking it.
