The starting hunch

"This wants to travel together"

A handful of skills that compose, a script that does what the model can't, conventions tying them together. When a set of parts feels like it should be installed as one thing — that's the smell of a plugin.

The instinct was right. The interesting part is what happens to the glue — the CLAUDE.md that unified everything.

The reframe

A plugin is a crate, not a brain

A plugin doesn't do anything. It bundles components that do the work, and a manifest ? that just labels the box.

Your "like a Project, but more formalized" instinct was exactly right: a plugin is a project's parts, sorted into named slots and given a shipping label so it can travel.

The crux

Always-on vs. loaded-when-triggered

CLAUDE.md — always loaded ?

Perfect for rules that must always hold — lifecycle, a Definition-of-Done gate. But it's not a plugin component.

a skill — lazy, triggered ?

Ships cleanly in a plugin. But a gate that only fires when triggered is useless — the one time it doesn't fire is the time something slips.

The special thing about CLAUDE.md is its loading, not its content — and the menu has no always-on slot.

The resolution

Make the plugin a factory

Don't bundle always-on context. Install it. A plugin that plants a project CLAUDE.md wins for free — a project's CLAUDE.md is always-on by nature.

skills/lesson
the orchestration logic (lazy-loaded)
templates/CLAUDE.md
the always-on copy, planted into each project
skills/start
the installer that stamps the folder out
The plugin is the factory; the live workflow is the project it stamps out.

A boundary that matters

The gate splits across the line

  • Code in the plugin.
    — installed once, shared by every project → it must stop knowing where it lives (resolve the root, don't assume cwd).
  • State & wiring in the project.
    — progress.json is per-project data; a git hook lives in .git/hooks. Neither can sit in a shared install.
  • Two different "hooks".
    — a git hook ? ≠ a Claude plugin hook ?.

The gate, redesigned

A git-agnostic Stop hook

Gate on auditable artifacts, not commits. A Stop hook ? refuses to finish while a lesson is marked past its artifacts.

  • No-ops outside its own projects — a plugin hook fires globally, so it must do nothing when there's no project to gate.
  • Blocks only real violations — a dedicated read-only --gate mode, separate from the noisier --check.

What bit us (live)

Four findings worth keeping

  • Code coupled to its location — progress.mjs trusted import.meta.url; broke in a plugin until it resolved the root.
  • Plugin hooks fire everywhere — a naive gate would block any session in any repo.
  • Don't declare standard paths — listing hooks/hooks.json in the manifest double-loads it ("Duplicate hooks file").
  • Mounted-FS git locks — stale .git/*.lock blocked the next commit until removed.

Key takeaways

What you now know

  • A plugin is packaging — a fixed menu of components plus a label. It doesn't act; its parts do.
  • The menu has no always-on slot → a plugin installs a project CLAUDE.md instead.
  • Moving code into a plugin means it must stop knowing where it lives; hooks fire globally.
  • One repo serves both install paths — and a plugin is just files, so it can build itself.
