Where we are

Ingestion is the first stage — and errors here cascade the furthest

101 left us a runnable skeleton with a deliberately crude ingest: fixed-size windows. The problem showed up immediately on the Monster Manual "G" page.

Fixed windows split across monster boundaries — a single chunk straddling the end of Githyanki and the start of Goblin. That pollutes the embedding and the citation.
The cascade (from 101): you can't prompt your way to context that was never cleanly retrieved. Garbage chunks are an upstream error — everything downstream inherits them.

So ingest is the highest-leverage place to get the units right. Fix the chunk, and retrieval/generation get a clean substrate.

RAG 102 · Why ingest first

The central design decision

"What is a chunk?" is a precision ↔ completeness tradeoff

Get this wrong and no embedder or reranker can save you: too small and the answer is scattered; too large and the real signal drowns in noise.
RAG 102 · What is a chunk

The design space

Three ways to chunk — what each optimizes

Fixed-size

N tokens, slide

Dead simple, cheap, deterministic. Ignores meaning — happily cuts mid-sentence and across entities. What 101 used.

Structure-aware

Split on the DOM

Boundaries come from the document's own headings. One monster = one chunk. Cheap, deterministic, and the heading is the citation. Our choice.

Semantic ?

Model finds shifts

Cut where meaning changes. Highest recall in benchmarks — but embeds every sentence at ingest. Expensive.

Why structure-aware here: D&D Beyond pages are rigidly templated, so the structure already is the semantics — we get most of semantic chunking's benefit at fixed-chunking's cost.
RAG 102 · Three strategies

The evidence, honestly read

Ground boundaries in structure — not in a magic token count

Verified

  • Semantic chunking out-recalls fixed splitting (Chroma, ~0.91 vs ~0.85) — vendor, single benchmark, so directional not portable.
  • The win comes from boundaries that respect meaning — which structure also gives us, cheaply.

Killed as folk wisdom

  • "Optimal chunk size is 200–500 tokens" — no primary source.
  • "512 tokens, 10–20% overlap is the default" — no controlled benchmark.
So we don't pick a number — we pick a boundary rule (the heading), and treat size as a guard to tune later (107).
RAG 102 · Evidence vs folk wisdom

The design, built

One semantic unit per chunk — size is a guard, not the boundary

  • A chunk = one leaf heading section — for the Monster Manual, one monster. Boundaries parsed from the DOM heading hierarchy.
  • Size guard, not size rule: an oversized section sub-splits recursively under a generous max; a trivially tiny fragment merges into its parent. The numbers are guards to tune in 107, never magic constants.
  • Atomic blocks stay whole — a stat block is never cut in half, even if the size guard fires around it.
Result on the "G" page: 49 sections → 48 clean chunks — one intact chunk per monster, plus a "group overview" chunk where a heading carries its own lore (e.g. Goblins).
RAG 102 · The chunking rule

Metadata, for near-free

The heading you split on is the citation

Structure-aware chunking hands us metadata as a side effect. We populate the meta bag the 101 schema already reserved — no migration:

  • source — the book (e.g. "Monster Manual")
  • breadcrumbs[] — the heading trail
  • chunkId — stable, deterministic id
  • kind — statblock / prose / mixed
Breadcrumbs do double duty: the joined trail Monster Manual › G | Monsters › Goblins › Goblin is the human-readable citation and a future filter key for hybrid retrieval (104).

Deferred on purpose: typed entity fields (CR, creature type) and a normalized monster DB — non-breaking to add later, since meta is a JSON bag. Baselines first.

RAG 102 · Metadata sidecar

What the build actually taught us

Two assumptions the real DOM falsified

1 · There are no tables. The spec said "keep tables whole." But DDB renders stat blocks as <div class="…stat-block-background"> — zero <table> elements on the page. A "don't split a table row" rule would have matched nothing.

Lesson: "structure-aware" means the page's structure — which you have to actually inspect, not the structure you assumed.

2 · 101's "idempotent re-ingest" was a myth. sqlite-vec's vec0 table ignores INSERT OR REPLACE, so re-ingesting into a non-empty DB crashed. It only ever "worked" against an empty DB — i.e. it was never tested.

Lesson: an "idempotent" claim you never re-ran is a claim you haven't tested. Fixed with a DELETE-then-INSERT.

The bigger picture

Chunking choices ripple all the way to citation quality

  • Downstream reach: chunk boundaries shape what can be retrieved, what a reranker sees, what the LLM is grounded in, and what a citation can point at. Get them wrong upstream and every later stage inherits it.
  • No portable "best" size: it's corpus- and task-dependent — so we set a boundary rule now and measure the guard numbers on our own corpus in 107.
  • Why D&D is generous: rigid, standardized structure makes structure-aware chunking unusually clean — the page hands us boundaries and citations for free.
RAG 102 · Bigger picture

Key takeaways

What to walk away with

  • The decision: "what is a chunk?" — the smallest unit that still holds a whole answer (precision ↔ completeness).
  • The rule: boundaries from structure, size as a guard (sub-split / merge), atomic blocks kept whole.
  • Free metadata: the heading trail is both citation and future filter key — no schema change.
  • Inspect reality: there were no tables; an untested "idempotent" claim was false. Build against the real DOM, not the assumed one.
  • Clean blame: 102 fixed chunk content; neighbor ranking is 103's embedding job.
