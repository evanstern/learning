The one-liner

MCP is a standard protocol that lets any LLM app talk to any tool or data source

Think of it as USB-C for AI applications: one connector spec, so a model-driven app and an external capability can interoperate without bespoke, per-pair integration code.

Today's path: the problem → why now → the three primitives → the architecture → transport & lifecycle.
MCP 101 · What is it

The problem

Before a standard, every tool was bespoke glue

You could already wire tools into a model with plain function-calling. So why a protocol?

  • Nothing was portable. Each app integrated each tool its own way.
  • "I like what your tool does, but I have to re-translate how it does it to fit my app."
  • Every client × every tool = a fresh, hand-maintained integration.
The pain in one word: re-implementation. The ecosystem kept rebuilding the same connections, one client-tool pair at a time.

Note: this is the same reason we standardized on USB, HTTP, and language servers (LSP). Sound familiar?

MCP 101 · The problem

Why a standard matters

M×N bespoke integrations collapse to M+N

Integration cost goes from O(n²) to O(n). Quadratic kills ecosystems; linear grows them. That gap is "portability."
MCP 101 · M×N → M+N

Why now (the deep reason)

The "client" stopped being a developer — it became the model

O(n²) integration pain isn't new. Drivers, plugins, and APIs have had it forever. So why did this standard erupt for LLMs?

  • Old world: a human dev wires a known integration once, documents it, done. Ad-hoc is fine.
  • New world: an autonomous model does "everything everywhere," and you can't hand-script every path in advance.
Consequence: tools must describe themselves — names, descriptions, JSON schemas the model reads at runtime to decide how to use them.

MCP is built for a consumer that discovers and reasons about tools on the fly, not a dev reading docs and hardcoding calls.

Three primitives the server exposes — each triggered by a different actor

Model-controlled

Tools

Server lists them; the model decides to call one. Actions & side effects.

tools/list · tools/call

App-controlled

Resources

Server lists readable data; the host app decides what to read into context — not the model, not the user.

resources/list · resources/read

User-controlled

Prompts

Server lists templates; the user fires one — a slash command like /review-pr.

prompts/list · prompts/get

All three are capabilities one MCP server can expose — the split is who triggers each. A filesystem server might expose your files as resources (the app reads them), write_file as a tool (the model calls it), and /summarize as a prompt (the user fires it).

MCP 101 · Architecture

Reading the architecture

The server is "dumb" on purpose — that's the point

  • Host owns the brain (the LLM) and the trust/permission decisions.
  • Client = a per-server socket inside the host; one client per connected server.
  • Server holds tools + data and never talks to the model — it just answers MCP requests.
Why "dumb" is good: your server has no idea a model exists, so it's portable across any host — and the host stays the single place that polices what reaches the model.

We pick this thread back up in 102, where that boundary becomes a security story.

MCP 101 · The split

Transport

Two transports — the same protocol rides on top

Local

stdio

Host launches the server as a subprocess; they talk over stdin/stdout. Your focus and the-stacks.

Remote / multi-user

streamable HTTP

Server runs elsewhere; POST + SSE stream. Hosted connectors.

Both carry JSON-RPC 2.0 messages. The transport is just the pipe; the protocol is the language. (The wire format is its own deeper sub-topic.)

MCP 101 · Lifecycle

The design decision that matters most

Capability negotiation buys you survival

At initialize, both sides declare versions and which features they support. Features like resources, prompts, sampling are optional — used only if both negotiated them.

  • A new client can talk to an old server, and vice versa.
  • Both sides degrade gracefully instead of crashing on an unknown method.
The real win: client teams and server teams upgrade on totally independent schedules. No "flag day" where everyone must upgrade at once.

For an ecosystem of thousands of independently-owned servers, that's the difference between living and dying. Same trick browsers use to evolve HTTP.

MCP 101 · Capability negotiation

One-slide mental model

MCP is a self-describing, negotiated contract between a model-driven host and dumb, portable servers

Tools describe themselves so the model can choose them at runtime; the host holds the brain and the trust; servers stay ignorant of the model so they plug in anywhere.

MCP 101 · Mental model

Key takeaways

What to walk away with

  • Why it exists: portability — implement once, plug in anywhere. O(n²) → O(n).
  • Why now: the consumer became the autonomous model, so tools must self-describe for runtime discovery.
  • Three primitives sorted by who pulls the trigger: tools (model), resources (app), prompts (user).
  • Architecture: host owns the LLM + trust; one client per server; servers are dumb & portable.
  • Lifecycle: capability negotiation lets a decentralized ecosystem evolve without a flag day.
