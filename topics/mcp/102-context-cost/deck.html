Where 102 begins

Recap: the picture from 101

  • Portability: a standard protocol collapses M×N integrations into M+N.
  • Self-describing tools: the consumer is the model, so tools advertise themselves for runtime discovery.
  • Architecture: host owns the LLM + trust; one client per server; servers are dumb & portable.
That self-describing, connect-everything design is exactly what creates 102's problems. Every nice property has a bill.
MCP 102 · Recap

The hidden tax

Connecting a server isn't free — it spends context

At connect time, every server dumps its full tool catalog — names, descriptions, and input schemas — into the model's context window, before a single tool is called.

The model has to "hold the menu in its head" to know what it can do. A big menu is a standing cost on every turn.
MCP 102 · Context cost

Two failures fall out of that

Token bloat and choice overload

The obvious one

Token bloat

16 verticals × ~15 tools × a schema each = potentially hundreds of tools, tens of thousands of tokens gone up front. Less room for the task, higher cost, more latency.

The sneaky one

Choice overload

Even if the window holds it: the model must pick one tool from a giant menu. With 300 near-duplicate options across verticals (four searchs, three get_customers), decision quality collapses — wrong tool, wrong vertical, or freeze.

Token bloat is the cost you can see. Selection degradation is the one that quietly wrecks reliability.
MCP 102 · Token bloat & choice overload

The mitigation

The gateway / "librarian" + tool search

Model asks for what it needs → gateway searches all catalogs (tool RAG) → returns just the few relevant tools. Small context, sharp choices. (This very session uses deferred tools + a ToolSearch step.)
MCP 102 · Gateway

Every fix buys a new failure surface

What the gateway costs you

Org

Ownership burden

Someone must build & maintain it, and keep it aware of every vertical's catalog. A Conway's-Law problem as much as a technical one.

Addressed by → run it as a platform product: each vertical team owns its server behind a published contract; a platform/CoE team owns the gateway + registry that auto-discovers catalogs.

Availability

Single point of failure

Gateway down = all 16 verticals unreachable at once. Plus an extra hop: latency and a throughput chokepoint in the hot path.

Addressed by → standard distributed-systems hygiene: replicate/load-balance, health checks, timeouts & circuit breakers, cache catalogs, and degrade gracefully (cached or direct connections).

Correctness

Silent mis-routing

If search returns the wrong tools — or misses the right one — the model can't recover, because it never learns the correct tool existed. A visible failure becomes a silent one.

Addressed by → treat the router like a model: eval sets & offline benchmarks, full logging/observability, generous top-k, and a confidence fallback to a broader listing.

None of these are unique to MCP — they're known distributed-systems and MLOps playbooks. We flag the solution shape here and save the deep dives for their own sessions.
MCP 102 · Gateway costs

The heavyweight failure mode

Prompt injection via tool output

Tool results flow back into context. A get_ticket(id) returns text typed by a customer.

"Disregard all previous instructions and …"

The model reads its whole context as one stream. It can't cleanly tell data it fetched from instructions to obey.

Root cause: data and instructions share one channel. There is no reliable in-band marker that separates "content" from "command."

You cannot prompt your way out of this with "ignore malicious instructions" — that's just more in-band text an attacker can talk over.

MCP 102 · Injection

Why the mesh makes it catastrophic

The confused deputy with the keys to everything

An agent's effective privilege = the union of every tool it can reach. Through the gateway, the support-ticket reader can also touch Billing's refund_customer and CRM's export_contacts.

Low-trust input (a box any customer can type into) + high-privilege reach = a single poisoned ticket pivoting to effective root across the entire portfolio. The gateway that solved context also concentrated all authority into one compromisable decision-maker.
MCP 102 · Confused deputy

A common wrong turn

Why RBAC doesn't touch this

RBAC governs identity: who are you, what may you do.

Injection is a provenance / integrity problem: a fully-authorized agent is tricked by instructions laundered through data.

The agent has every right to call refund_customer. It's just being told to by the wrong author. RBAC answers "may this agent?" — not "who really asked?"

Different axis entirely. That's why authz alone can't save you here.

MCP 102 · RBAC vs provenance

The defensive stance

You can't reliably stop injection at the input

So stop trying to win that race. Instead, constrain what a compromised agent can do.

Containment > prevention.

MCP 102 · Containment

Defense levers · lower privilege

Scope the agent · gate the action

No god-agent

Scoped, task-specific agents

The agent that reads tickets simply does not hold refund_customer. Separation of duties. The dangerous union only exists if you build it that way.

Un-injectable

Deterministic gates + caps

The model can't call refund_customer directly; it proposes, and plain non-LLM code enforces rules: if amount > 50: deny/escalate. You can't prompt-inject an if.

MCP 102 · Scope & gate

Defense levers · break the autonomy chain

Human-in-the-loop, wired into orchestration

The model proposes the privileged action; a human commits it. (Exactly what "allow this tool?" prompts do.) Cheapest, most effective circuit-breaker.

Compose it: the deterministic gate auto-approves the safe, bounded cases and only escalates the ambiguous ones to a human.
In practice

Gates = graph edges

In LangGraph, a gate is a conditional edge (plain code) choosing the next node. In n8n, an IF/Switch node. Agents are nodes; the flow between them is code the model can't redraw.

Rule: branch on structured, trusted state (a computed amount), never on the model's free-text say-so — or the injection leaks back in one layer up.

MCP 102 · Human-in-the-loop

Defense levers · architectural quarantine

The dual-LLM (planner / executor) pattern

Poisoned prose reaches the parser, never the planner or the hands. Instructions can't ride inside a typed number.
MCP 102 · Dual-LLM

Trace it

Where the attack dies

  • Injected "refund $9,999" reaches only the quarantined parser — the planner never receives it as text.
  • So the attack can't command anything; at worst it produces a bad value (amount: 9999), arriving tainted.
  • That tainted value hits the deterministic policy engine: 9999 > cap → deny / escalate.
Key property: a fooled LLM can produce a bad value, never a bad action — because actions are gated by code the prose can't touch. Blast radius is bounded by your policy, not by how gullible the model is.
MCP 102 · Trace

The mechanism underneath

Taint tracking = information-flow control

Sources are labeled at design time. A tool's definition declares its trust level: get_ticket → untrusted; get_balance → trusted.

The harness tags at the boundary. Data crossing in from an untrusted source is stamped tainted automatically — no LLM judgment involved.

Taint is sticky. Anything derived from tainted data inherits the tag: summarize it, extract a field, concatenate it — still tainted. A value is trusted only if every input that fed it was trusted.

This is the same technique security tools use to trace untrusted input for SQL-injection / XSS — ported onto LLM data flow.

MCP 102 · Taint

How does the planner know to quarantine?

It doesn't decide — provenance does

If the planner had to judge "this looks untrusted," that judgment would be the exact thing injection targets. So the decision is taken away from the LLM.

  • The planner emits a plan/program; routing to the quarantined LLM is a consequence of the taint tag, enforced as that plan runs — not a planner choice.
  • It writes "summarize the ticket"; the executor sees the value is tainted and dispatches that step to the tool-less parser automatically.
What is "the executor"? The deterministic runtime — plain code in the trusted orchestration layer, the same place the policy engine lives — that runs the planner's emitted plan, carries taint per value, and gates each step. It is not an LLM. (In LangGraph terms: the graph executor running your nodes & conditional edges.)
The one human decision: which sources are untrusted. Mislabel a source (an "internal" wiki that contractors can edit) and the machine faithfully ignores a hole.
MCP 102 · Provenance routing

A subtlety worth nailing

"Trusted" ≠ "safe"

Trusted = the principal whose authority the system acts under — bounded by their own permissions. Your authenticated request goes straight to the planner. That's correct.

Injection is the crime of a non-principal (a ticket author) smuggling directives through data to borrow your authority.

Governing principle: an agent must never hold privileges greater than the trust level of whoever can drive it.

MCP 102 · What "trusted" means

The crux: how do you cover the red flags?

Default-deny by capability — not red-flag detection

Don't enumerate "all the bad things" — that's the blocklist trap, the same losing game as detecting every injection.

Flip it: allowlist. Ask "is this exact action, with data of this provenance, explicitly permitted?" If not → denied.

Security boundary = completeness of the allowlist, not cleverness of the detector. Default-deny is tractable; "detect all evil" is not.

This is capability-based security — and the headline idea in DeepMind's CaMeL ("CApabilities for MachinE Learning"): defeat injection with classic software security, not more AI.

MCP 102 · Default-deny / CaMeL

What it looks like in code

A CaMeL-style plan — the refund scenario

# P-LLM plans from the TRUSTED request —
# it never sees the ticket text.
ticket = get_ticket(123)        # ☣ tainted
refund = query_quarantined_llm(  # Q-LLM, no tools
    "Extract the refund from [ticket]",
    output_schema=RefundRequest,
)                                # ☣ inherited
issue_refund(               # privileged tool
    order_id=refund.order_id,
    amount=refund.amount,        # ☣ tainted
)

Not a new language — a locked-down subset of ordinary Python (parsed via ast). Safety is in the interpreter + capabilities, not the syntax.

Policies the executor enforces per call:

get_ticket → always allowed
query_quarantined_llm → always allowed (holds no tools)
issue_refund → only if amount ≤ 50 and the order belongs to the ticket's customer; else escalate to a human
Injected "refund $9,999" → at worst a tainted amount=9999 → fails the issue_refund policy → escalated. The attack dies in code, not in the model.

Bringing it home

Mesh-scale design: open questions for ~16 verticals

  • Granularity: one server per vertical? per capability? per data domain? ("16" may be the wrong number.)
  • Taming many servers: gateway + tool search, namespacing to avoid tool-name collisions, a registry.
  • Orchestration & trust: where do cross-vertical workflows live, and how is provenance tracked across them?
  • Conway's Law: the ownership model is the real constraint — central platform team vs. each vertical owning its own server (with the MCP contract as the team boundary).
MCP 102 · Mesh-scale design

Key takeaways

What to walk away with

  • Context isn't free: connecting servers costs tokens and degrades tool selection.
  • Gateways + tool search tame it — but add ownership, a single point of failure, and silent mis-routing.
  • Injection via tool output + the union-of-privileges mesh = confused deputy with effective root.
  • RBAC misses it (identity vs provenance). The answer is containment over prevention.
  • Defense by design: scoped agents, deterministic gates, human-in-the-loop, dual-LLM quarantine, sticky taint, and default-deny capabilities.
