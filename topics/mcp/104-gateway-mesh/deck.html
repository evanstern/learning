1 · Granularity

The server cut is a least-privilege boundary

Per-vertical recreates the god-agent / union-of-privileges problem at packaging time. Cut by capability/task instead.
MCP 104 · Granularity

Why the capability cut wins

Capability-scoped servers are trust-homogeneous

"Bind billing-refunds" now means "grant one trust tier" — everything inside is the same risk class. The packaging unit and the trust boundary snap into alignment. Per-vertical smears tiers together; per-capability separates them.

MCP 104 · Trust-homogeneous

Collisions & namespacing

Duplicate names don't error — they vanish

Billing, CRM, and Shipping each name a tool search. All land in the gateway's catalog.

A naive gateway silently shadows one with another (last-registered wins) — a whole tool becomes invisibly unreachable, with no error. Best case, the model mis-selects between look-alikes.
Fix: namespace by server. billing.search, crm.search. Restores uniqueness, gives the model a disambiguating signal — and the prefix encodes the owning server (matters for ownership later).
MCP 104 · Collisions

The registry

MCP discovers tools — not servers

The handshake (tools/list + the list_changed notification) tells the gateway what an already-connected server offers.

But it never tells the gateway a new server exists. Discovering servers = config / a registry you build and run. The protocol gives you the calling; not the directory.

So the registry owns:

  • which servers exist + where to reach them
  • owners and trust levels
  • versioning + a reviewed contract — the danger is silent schema drift breaking live agents
MCP 104 · Registry

2 · Availability

The gateway turns independent failures into one correlated failure

Before: a Billing crash cost you Billing. After: the gateway sits in the hot path of all 16, so one gateway hiccup makes every vertical unreachable at once — including the healthy ones.

"The mothership falls and the whole swarm goes dark." Centralization bought portability and paid for it in blast radius.
MCP 104 · Availability

Hardening the mothership

The slow vertical is scarier than the dead one

Kill the SPOF: run N stateless gateway replicas behind a load balancer + health checks. Any replica serves any request.

The trap: a slow vertical (not cleanly down) makes calls pile up and exhaust the gateway's pool — recreating the full-mesh outage from one sick drone.

Restore fault isolation at the gateway:

  • Timeouts — never block forever on a vertical
  • Circuit breakers — after repeated failures, fail fast instead of hammering
  • Bulkheads — cap concurrency per vertical so one can't eat all capacity
  • + serve catalogs from cache; return a clean error so agents route around
MCP 104 · Hardening

The extra hop

The network is cheap; the gateway's own work isn't

Every call is now client → gateway → vertical: added nodes/edges = inherent latency and a new fault point.

Wire levers: connection pooling / keep-alive, streaming passthrough (no buffering), co-location.

The real cost is usually the gateway's per-request work — if it runs a semantic tool-search on every call, that compute dwarfs the network. Biggest win: cache / precompute the routing.

3 · Router quality

Silent mis-routing: healthy, but wrong

The gateway's tool-search decides which 2–3 tools the model even sees. It's an ML component, not plumbing.

A crash is loud (error, alert). Mis-routing is silent: 200 OK, fast, tidy — and the wrong tools, or the right one omitted entirely so the model can't recover. It passes every availability check while being wrong.
MCP 104 · Silent mis-routing

Catching a healthy-but-wrong system

Evaluate the router like a model, not like code

  • Eval set: representative queries paired with the tool(s) that should win.
  • Measure a rate, not pass/fail: did the right tool land in the top-k? (recall@k) — it's a ranker.
  • Regression gate: re-run on every router/embedding change or new vertical (adding servers can quietly degrade routing).
  • Observability: log query → candidates → chosen → outcome; mine real misses.
  • Feed misses back: the eval set is a living asset.
Health checks answer "is it up?" — never "is it right?" Availability monitoring is blind to a correctness failure.

4 · Ownership (Conway's Law)

The central team paves the road; verticals drive on it

Who builds the 16 servers? Not one central team (a bottleneck that doesn't know 16 domains). The scarce knowledge is the domain, not MCP.

"Easier to teach the Billing team MCP than to teach me Billing." Push implementation to the domain experts.

Central platform team owns the paved road:

  • MCP SDK / template, namespacing & schema standards
  • the gateway, registry, eval harness
  • the 102/103 security library (taint + gate) as defaults

Decentralizing requires this governance — or you get 16 incompatible styles.

MCP 104 · Ownership

Conway's Law, turned intentional

The MCP interface is the contract between teams

However you split the teams, the architecture mirrors it — so make the split deliberate. The versioned MCP contract becomes a clean, explicit boundary: vertical teams own their servers, the platform team owns the road, and the schema is the handshake between them.

MCP 104 · The contract

Auth across verticals

Zero trust — the gateway is not a trust anchor

Don't let the gateway call everything with its own god credentials: (1) it becomes a single point of total compromise; (2) every vertical sees one omnipotent caller — least-privilege gone, the confused-deputy rebuilt.

Do, instead:

  • Identity propagation (on-behalf-of): authorize against the end principal, not the gateway
  • Per-vertical scopes: read-only in CRM, refund-only in Billing — never a blanket pass
  • Each vertical enforces its own authz — never "the gateway said so"

Effective power = (principal may) ∩ (vertical permits), enforced at the vertical.

5 · Bringing it home

The gateway is a scaling tool, not a starting point

Don't build a gateway + 16 servers on day one. The gateway earns its complexity only when many servers cause context bloat. With 2–3 servers, an agent connects directly.

Wedge in:

  • Find one painful cross-vertical workflow a human does today by hand.
  • Build only the tools that workflow needs (capability cut), wired straight to one agent — no gateway/registry/router yet.
  • Ship it, measure the time/$ saved. That's your evidence to fund the rest.
MCP 104 · Migration

The wisdom slide

When not to build the mesh

  • You can't fund the run cost (not just the build). A half-maintained mesh is worse than none.
  • No real cross-vertical demand — if workflows rarely span verticals, point-to-point integrations win.
  • Not enough scale — the gateway/router/registry only pay off with many servers; below that it's pure complexity tax.
  • The org can't own servers — without vertical-team ownership, you're back to the central bottleneck.
The test: if you can't name a painful, repeated, cross-vertical workflow with real ROI, you don't have a reason yet. Don't build for an imagined future.

Key takeaways

What to walk away with

  • Granularity is a least-privilege boundary — cut by capability (trust-homogeneous), not vertical.
  • Namespace to stop silent tool-shadowing; run a registry (MCP discovers tools, not servers).
  • Harden the gateway (stateless replicas) and restore isolation with timeouts + circuit breakers + bulkheads.
  • Evaluate the router like a model — mis-routing is silent; availability checks won't catch it.
  • Pave the road, don't build all 16; propagate identity (zero trust); wedge in, and don't build without ROI + run budget.
